# BurnRate — AI Spend Audit Tool

BurnRate is a free web app that helps startup founders and engineering teams identify where they're overpaying on AI tool subscriptions. You enter your tools, plans, and monthly spend — the audit engine runs hardcoded rules against known pricing, flags inefficiencies, and shows exact savings with an AI-generated summary.

Built as a Credex assignment submission.

---

## What It Does

- Accepts input for 8 major AI tools (Cursor, GitHub Copilot, Claude, ChatGPT, Gemini, Windsurf, Anthropic API, OpenAI API)
- Runs a deterministic audit engine that checks plan efficiency, seat count vs team size, and overspend patterns
- Generates a 2-sentence AI summary via Gemini 2.0 Flash
- Stores results in Supabase and generates a shareable URL
- Captures leads via an email form with honeypot + rate limiting

## Running Locally

**Prerequisites:** Node.js 18+, a Supabase project, a Gemini API key

```bash
git clone https://github.com/your-username/burnrate
cd burnrate
npm install
```

Create `.env.local` in the project root:

```
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=your-publishable-key
GEMINI_API_KEY=your-gemini-api-key
```

Run the Supabase SQL in your project dashboard:

```sql
create table if not exists audits (
  id uuid default gen_random_uuid() primary key,
  created_at timestamp with time zone default now(),
  tools jsonb not null,
  team_size text,
  use_case text,
  total_monthly_spend numeric,
  total_monthly_savings numeric,
  total_annual_savings numeric,
  is_high_savings boolean,
  is_optimal boolean,
  recommendations jsonb,
  ai_summary text
);

create table if not exists leads (
  id uuid default gen_random_uuid() primary key,
  created_at timestamp with time zone default now(),
  audit_id uuid references audits(id),
  email text not null,
  company text,
  role text
);

alter table audits enable row level security;
alter table leads enable row level security;

create policy "public insert audits" on audits for insert with check (true);
create policy "public select audits" on audits for select using (true);
create policy "public insert leads" on leads for insert with check (true);
```

Then start the dev server:

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

## Trade-Off Decisions

**1. Deterministic audit engine over AI-generated recommendations**

The audit rules are hardcoded against known pricing data, not generated by an LLM. This means recommendations are consistent, testable, and defensible — two identical inputs always produce the same output. The trade-off is maintenance cost: pricing changes require manual updates. For a trust-sensitive product like spend advice, consistency matters more than flexibility.

**2. Gemini over Claude for the AI summary**

Gemini 2.0 Flash is free on the API, Claude Haiku costs money. For a 2-sentence summary that is cosmetic rather than functional, the quality difference is negligible and the cost difference is real. If the summary call fails, the results page falls back to a hardcoded template — the core product never depends on this call succeeding.

**3. Supabase over a serverless DB like PlanetScale**

Supabase gives Postgres, auth, row-level security, and a JS client in one free-tier package. For a project that only needs simple inserts and selects across two tables, this is significantly lower overhead than managing a separate auth layer and database connection pool. The trade-off is Supabase's cold start latency on the free tier.

**4. No user authentication**

Audits are anonymous and shareable by URL. Adding auth would reduce friction loss on the form dramatically — every sign-up gate increases drop-off — and the lead capture modal after the results page collects email voluntarily. The trade-off is that we cannot let users revisit their own past audits without the URL.

**5. In-memory rate limiting over Redis**

The capture-lead route uses a JavaScript Map for rate limiting (3 requests per IP per 10 minutes). This resets on every server restart and does not work across multiple serverless instances. A Redis-based solution like Upstash would be correct for production. For a hackathon submission with low traffic, in-memory is sufficient and avoids another external dependency. The decision is documented so reviewers know it is intentional, not an oversight.
